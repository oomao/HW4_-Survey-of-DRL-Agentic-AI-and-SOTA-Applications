一、Why an AI Harness？

研究生為新 DRL 主題做文獻綜述時的三個痛點：

  1. LLM 知識截止 — 不知道 2025–2026 新 paper。
  2. 幻覺引用 — LLM 常編造看似真實的 arXiv ID。
  3. 無 stateful 規劃 — 無法把大查詢拆成多階段並彙整。

AI Harness 用「LLM as controller + 外部工具 + memory + orchestration」解這三點。

二、系統設計（一張圖看完）

2.1 LLM Controller — 四個 sub-role（共享同一 model）

角色輸入輸出
① Planneruser query2–4 sub-topics + dependency DAG（JSON schema）
② Reasoner (ReAct)sub-topicThought / Action / Observation 循環，單 step 一個 action
③ Critic累積 notesbinary check + 缺失列表（max 1 re-search round）
④ Compilernotes + citations結構化 Markdown 報告（IEEE refs）

2.2 四個 Tools（≥3 ✓）

ToolSignatureSide effectFailure mode
arxiv_search(query, year_min, year_max, max_results) → list[paper]None無結果 → empty list；網路錯 → offline cache
paper_summarize(arxiv_id) → {contribution, methods, results, limitations}None未知 ID → {"error": ...}（不 raise）
citation_format(paper, style ∈ IEEE/ACM/APA/BibTeX) → strNone（純函式）缺欄位 → <unknown> placeholder
note_save(topic, content, tags?) → ackWrite notes.jsondisk error → {persisted: false}

2.3 Memory（雙層）

Short-term = LLM context window；Long-term = JSON note store（per-topic）+ citation DB（per-paper）。Compiler 只從 long-term memory 讀 facts，不接受 LLM inline 編造 — 這是幻覺防護的核心。
